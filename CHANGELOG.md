## [Unreleased]

### Phase 8 — PWA & Offline Support

- **Schema decision (option A): added `syncedFromOffline: boolean` and `clientId: string | null` to all five log models.** PRD §4.3 originally only declared `syncedFromOffline` on `FoodLogEntry`, but F-PWA-5 ("entries flagged `syncedFromOffline=true`") and F-PWA-4 ("all log creation forms work offline") want symmetric behaviour across food / exercise / water / sleep / measurement. Flagged the ambiguity, user picked option A. The `clientId` field is the new one — a UUID stamped by the client on offline-origin entries; combined with a sparse-unique compound index on `(userId, clientId)` (partial filter `clientId: {$type: "string"}` so the index is empty for live writes), it makes the replay endpoint race-safe via Mongo's E11000 instead of an app-level dedup table that would need its own consistency story.
- `next-pwa` integration in `next.config.mjs` (Workbox under the hood). Disabled in development so HMR isn't shadowed by stale precache hits. Runtime caching:
  - **API GETs → StaleWhileRevalidate** (`/api/.*`, GET) — last-cached dashboard renders instantly on relaunch; network refresh updates in the background. Capped at 256 entries / 24h.
  - **API mutations → NetworkOnly** (POST / PATCH / DELETE). We deliberately do NOT use Workbox's BackgroundSyncPlugin: it would replay raw `Request` objects, but our offline contract needs UUID-keyed idempotent batch replay so that a network glitch mid-flight cannot double-create. Workbox only handles the GET-cache half; the POST/PATCH/DELETE half is owned by `lib/offline/queue.ts` + `/api/sync/replay`.
  - **Page navigations → NetworkFirst** with 3 s timeout, falling back to the cache so the app shell still opens when offline (Phase 8 DoD).
  - **Static assets** (icons, manifest) → CacheFirst, 30-day TTL.
- `public/manifest.json` per F-PWA-1: `name`, `short_name`, `description`, `start_url: /dashboard`, `scope: /`, `display: standalone`, `theme_color: #1976d2` (matches MUI primary in `lib/theme.ts`), 192/512/512-maskable icons. Icons are generated by `scripts/generate-icons.mjs` (no image-lib dep — a hand-rolled PNG encoder that draws a flat blue rounded square with a centred white "C" glyph, with a 10% safe-zone padding on the maskable variant per the W3C maskable-icon spec). Re-run the script if the brand colour ever changes.
- `app/layout.tsx` declares `manifest: "/manifest.json"`, Apple `appleWebApp` metadata, icon links, and mounts the `<OfflineSyncOrchestrator />` and `<InstallPrompt />` once at the root. Both components render nothing visible until the relevant browser event fires.
- `lib/offline/queue.ts` — IndexedDB-backed sync queue per PRD §4.8 via the `idb` library. Schema mirrors §4.8 verbatim: `{id, type, payload, createdAt, retries}` with `id` as the keyPath and a `byCreatedAt` index for oldest-first drain order. Public API: `enqueue`, `listQueue`, `queueSize`, `dequeueMany`, `incrementRetry`, `clearQueue`, plus pure `nextBackoffMs(retries)` (1s, 2s, 4s, … capped at 5 min). Cached `dbPromise` is reset between tests via `__resetForTests()` so `fake-indexeddb` instances don't leak.
- `lib/offline/replay.ts` — `drainOnce` and `drainUntilEmpty`. Sends queued items in batches (default 50, max 100 per request) to `/api/sync/replay`; processes per-item results: `created` and `duplicate` both dequeue (the entry is on the server either way); `failed/retryable=true` bumps the row's retry counter so the next pass backs off; `failed/retryable=false` (validation error etc.) is dropped — keeping it around would block every future drain forever. Network-layer failure (fetcher rejects) bumps every item's retry counter without dequeuing. **401 is handled specially**: the queue is left intact and retries are NOT bumped (the data is fine, the session just expired; the user signs back in and replay resumes).
- `lib/offline/use-offline-mutation.ts` — the single `tryOnlineOrEnqueue` helper every log form uses. Decision tree: (1) `navigator.onLine === false` → enqueue immediately, no network attempt; (2) network attempt throws (`TypeError` from `fetch`) → enqueue + queued toast (browser thinks it's online but the request didn't reach the server, which happens on captive-portal Wi-Fi); (3) any HTTP response (even 4xx/5xx) → bubble back to the caller. We do NOT enqueue on 4xx because the payload is bad and replay would just fail again forever.
- `app/api/sync/replay/route.ts` — `POST /api/sync/replay`. Zod-validated batch (`{items: Array<{id: uuid, type: QueueType, payload: object}>}`, max 100 items per request, items use `.strict()`); auth-gated by `getSessionUser`; per-item dispatched to type-specific handlers that share validation with the live POST endpoints (`FoodLogCreateSchema`, `ExerciseCreateSchema`, `WaterCreateSchema`, `SleepCreateSchema`, `MeasurementCreateSchema`). All persisted entries set `syncedFromOffline: true` (F-PWA-5) and stamp `clientId` with the queue's UUID. **Idempotency** is enforced by the partial unique index — on E11000 we look up the existing entry by `(userId, clientId)` and return `{status: "duplicate", serverId}` so the client can dequeue it. Top-level 401/400 cover whole-batch rejections; per-item failures are reported in the 200-OK response so the orchestrator can dequeue the OK ones and keep the retryable ones. Food-log replays still run F-LOG-3 counter bumps (`food.timesLogged`, `food.lastLoggedAt`) — an offline-first user shouldn't have their food fall out of "Recent" just because they were offline at log time.
- `components/offline/sync-orchestrator.tsx` — root-mounted component that:
  1. Drains on mount (covers "user reopens the tab while online with stuff queued").
  2. Drains on every `online` event (resets in-session backoff counter to 0).
  3. Schedules backoff retries via `nextBackoffMs` after partial failures, scaled by an in-session `seenRetries` counter so a global outage doesn't hammer the server.
  4. Pushes a "you're offline" warning snackbar exactly once on `offline` events so the user understands why their next mutation says "queued".
  5. Renders one Snackbar at a time from the offline-toast Zustand store (`lib/offline/toast-store.ts`); MUI rotates them as the user dismisses.
  6. Invalidates the entire TanStack Query cache after a successful drain so any visible list/dashboard reflects the synced rows.
- `components/offline/install-prompt.tsx` (F-PWA-3) — captures `beforeinstallprompt`, suppresses Chrome's mini-infobar, and shows a Snackbar with an Install button. Dismiss is sticky for the session via `sessionStorage` so we don't pester the user. Renders nothing on Safari/Firefox (which don't fire `beforeinstallprompt`); their install paths go through the browser's own UI.
- **Critical wiring fix**: `app/providers.tsx` now sets `mutations.networkMode: "always"` on the QueryClient. TanStack Query's default is `"online"`, which **pauses mutations when `navigator.onLine === false`** — meaning `mutate()` would never invoke `mutationFn`, our `tryOnlineOrEnqueue` wrapper would never run, and offline writes would silently disappear instead of getting queued. We own the offline contract ourselves (enqueue → IDB → /api/sync/replay), so mutations must always run their fn regardless of network. Found while debugging the e2e replay test.
- The five log views now wrap their POSTs through `tryOnlineOrEnqueue`: `app/log/water/water-log-view.tsx`, `app/log/exercise/exercise-log-view.tsx`, `app/log/sleep/sleep-log-view.tsx`, `app/log/food/log-food-dialog.tsx` (create-mode only — edit-mode PATCHes don't queue, they just fail visibly because servings-only edits are rare and not a critical-path offline scenario), `app/measurements/measurements-form.tsx`. Delete and PATCH paths are intentionally NOT wrapped — F-PWA-4 covers creation only, and the conflict policy for mutating-existing-entry semantics is not in scope.

#### Migrations / indexes

- No new collections. The five existing log collections gained two fields and one index each:
  - **Schema additions** (all five): `syncedFromOffline: { type: Boolean, default: false }`, `clientId: { type: String, default: null }`. Both default-valued so existing documents keep working unmodified.
  - **New compound index on each**: `{ userId: 1, clientId: 1 }` with `{ unique: true, partialFilterExpression: { clientId: { $type: "string" } } }`. The partial filter means the index is empty for live (online) writes — those have `clientId: null` — and only carries entries that originated offline. So the index size scales with offline volume, not total volume; for a typical user it's empty. Created lazily on the first offline replay.
- New service-worker artefacts in `/public`: `sw.js` + `workbox-<hash>.js` are emitted by `next-pwa` on every `next build`. Both are gitignored — they're build outputs, not source.
- New `idb` database **on the client only**: `caloriespro-offline` (version 1) with a single `queue` object store and a `byCreatedAt` index. No schema migration required — `openDB`'s `upgrade` callback creates the store on first open.

#### Notes / decisions

- **Why not Workbox BackgroundSyncPlugin for the queue?** Workbox can serialise failed requests and replay them on `sync` events. But it replays raw `Request` objects, not idempotent operations — a transient failure between server-write and the response landing means the same UUID-less request gets re-sent and the server creates a duplicate. By owning the queue ourselves with client-generated UUIDs, we get true at-most-once semantics. We *also* benefit from being able to show "X queued" UX, batch the replay into one request instead of N, and provide a recovery story when the user signs back in (401 path keeps the queue intact). Workbox handles the GET-cache half of the SW; the POST-PATCH-DELETE half is ours.
- **Why a partial unique index instead of a separate replay-log collection?** A separate collection would need its own consistency story (write the dedup row → write the entry → if the entry write fails, undo the dedup row…) and either transactions (which require a replica set) or a careful state-machine. The partial unique index puts dedup on the data itself: `findOneAndUpdate` semantics aren't needed, the database rejects the duplicate at write time, the catch block looks up the existing row and returns `duplicate`. Race-safe even under concurrent retries from two tabs.
- **Why `networkMode: "always"` on mutations?** TanStack Query defaults to pausing mutations when offline so they retry when back online — but its retry mechanism replays the SAME mutation function which would just re-POST the same payload. That conflicts with our queue: we want the offline branch of `tryOnlineOrEnqueue` to run *immediately* and persist to IDB, not have the mutation paused indefinitely in memory (which would also be lost on tab close). `networkMode: "always"` makes the mutationFn run regardless of `navigator.onLine`, and our wrapper handles the routing.
- **Why poll IndexedDB in the e2e test instead of asserting on the toast?** The orchestrator pushes a "You're offline" warning toast on the `offline` event, BEFORE the user clicks anything. Then `tryOnlineOrEnqueue` pushes a "queued" info toast. Only one Snackbar renders at a time (head of the toast queue), so a `getByTestId("offline-toast").toContainText("queued")` would race against MUI's rotation. Polling IDB is authoritative — the queue write either happened or it didn't.
- **Backoff lives in two places.** Per-row retries (in `queue.ts`) influence per-item retry decisions; in-session retries (in `sync-orchestrator.tsx`) drive the *next-batch* schedule. The two are orthogonal: a single bad row doesn't slow down good rows, and a temporary server outage doesn't hammer at full speed once recovered. Both are pure functions of `nextBackoffMs(n)`.
- **Hidden DB-pending sentinel was tried but removed.** Initial draft rendered a screen-reader-only `data-testid="offline-queue-pending"` element when `hasPending`, but the orchestrator only refreshes that state after drains, not after enqueues by the wrapper. Wiring up a queue-changed event would have meant either polling the size or making `enqueue` push through the toast store — both more complicated than just polling IDB from the e2e. The IDB poll is the source of truth anyway.
- **Install prompt on Safari/Firefox.** Neither browser fires `beforeinstallprompt`, so our component renders nothing there. Users still install via the share menu (iOS Safari) or the page-info menu (desktop Firefox); we don't pretend to control that flow.
- **Lighthouse PWA score (NFR-2 ≥ 90) is verified manually**, not in CI. The DoD line stays unchecked until Phase 10 polish runs the full Lighthouse audit on a deployed build. The service worker is emitted, the manifest is valid, icons are present — all the required ingredients are in place.

### Phase 7 — Dashboard

- `lib/agg/totals.ts` — pure JS aggregation helpers (no DB, no React) so Phase 7 unit tests can pin behaviour without seeding Mongo. Covers `caloriesFromFoodEntries` / `macrosFromFoodEntries` (servings × snapshot fields summed across the day, with the nullable `fiberG`/`sugarG` from PRD §4.3 treated as 0), `caloriesFromExerciseEntries` / `waterFromEntries` (simple sums per F-EX-2 / F-WTR-2), `withinGoalRange` and `daysWithinGoal` (the F-DSH-5 ±10% band — a day with `0` calories does **not** count as "within goal of 0", a missing log is a missing day not a hit), `averageOf` (null-skipping mean used for sleep duration and quality), `latestSleep`/`latestWeight` (most-recent-with-data picks for the Today cards), and `weightTrend` (earliest+latest non-null pair across the window for the F-DSH-2 weight trend stat). Numeric outputs are rounded to 1 decimal place to avoid IEEE-754 fuzz like `0.1 × 100 + 0.2 × 100 = 30.000000000000004`.
- `lib/agg/pipelines.ts` — MongoDB aggregation pipeline builders (live in their own file so they're easy to lock with shape tests) plus thin runner wrappers. Each pipeline starts with a `(userId, date)` `$match` (project rule #4 — never trust the caller, always scope by userId), groups on `$date` (which is midnight UTC per PRD §4.3 so a calendar day is one bucket regardless of log time), and ends with a chronological `$sort`. Highlights:
  - **dailyFoodPipeline** uses `$multiply` of `snapshot.caloriesPerServing × servings`; `fiberG` / `sugarG` are wrapped in `$ifNull → 0` because Mongo's `$sum` is null-safe but `$multiply` propagates `null` and would corrupt the per-day total.
  - **dailySleepPipeline** sums `durationMinutes` and *averages* `quality`. PRD §4.6 doesn't forbid multiple naps for one wake date, so two entries on the same date roll up sensibly; in the common case of one-entry-per-night this collapses to "the row".
  - **dailyWeightPipeline** filters out `weightKg: null` (Phase 6 lets the user log only some metrics on a given day — DELIVERY_PLAN.md Phase 6 DoD), pre-sorts by `(date, createdAt)`, then `$last` to take the latest correction per day. Used for the trend chart and weekly weight stat.
- `lib/agg/dashboard.ts` — payload builders that combine the pipelines + pure helpers into the wire shapes the routes return. `buildTodayPayload` runs five aggregations in parallel via `Promise.all` (food, exercise, water, latest weight, latest sleep) plus a `User.findById` for the goal block; `buildSummaryPayload` runs five daily-grouped pipelines and then walks the requested window day-by-day so the client gets a row for *every* date (no chart-side gap-filling needed). Both use `startOfDayUTC`/`endOfDayUTC` from `lib/log/date.ts` (the same boundary helpers the existing log routes use) so a "May 1" payload covers exactly the same span as `/api/logs/food?date=2026-05-01`.
- API routes (Zod-style query validation, error shape `{error, details?}` per NFR-4):
  - **`GET /api/dashboard/today?date=YYYY-MM-DD`** — defaults to today's ISO date when `date` is omitted; 400 on a malformed date string.
  - **`GET /api/dashboard/summary?range=week|month`** — `range` is required and strictly enum-validated (so `?range=year` returns a 400, not a silently-clamped result); optional `date=` parameter pins the window's right edge (defaults to today). 7 days for `week`, 30 for `month`, both inclusive of the right edge.
  - Both endpoints reject anonymous callers with 401 via the standard `getSessionUser` guard. The middleware matcher already covers `/dashboard/:path*` from Phase 1; only the API endpoints need their own auth gate.
- Pages (gated by `withAuth` middleware — `/dashboard/:path*` was already in the matcher from Phase 1):
  - **`/dashboard`** — three-tab layout (`Today` / `This week` / `This month`) wired to `react/state` so the inactive tabs don't even mount their queries until selected. Server component does the auth redirect, then renders the client `DashboardTabs` shell.
  - **Today view** — six cards in a responsive `Grid` (mobile-stacked at 360px floor per NFR-6): calories in / calories out / net / remaining-vs-goal (with the F-DSH-5 chip when within ±10%), macros (P/C/F bars driven by `LinearProgress` against the user's per-macro goals), water (`LinearProgress` against `dailyWaterMl`), last sleep (formatted `7h 30m` with quality stars), current weight (the most recent measurement). Empty-state messages link directly to the relevant log page so a brand-new user doesn't see seven `—` placeholders.
  - **F-DSH-5 goal-met snackbar** — fires only on the *transition* from "not met" to "met" within a single page session (a refresh hours after the user hit the goal won't re-fire; that would be confusing). Tracked via a `useRef<boolean | null>` initialised to `null` so the very first render with `goalMet = true` is treated as the resting state, not a transition.
  - **Range view (week / month)** — top stat cards (avg daily calories across days with food logged, days within ±10% of goal, total exercise calories, avg sleep duration + quality, weight trend with start→end+delta, avg water on logged days) plus a `SummaryCharts` block: calorie line (consumed + burned with a dashed reference line at the goal), stacked macro bar (protein/carbs/fat in `g`), water bar with goal reference line, sleep duration bar with target reference line, weight line (skipped when no weight is logged in the window). Recharts is dynamically imported via `next/dynamic({ ssr: false })` so the Today tab doesn't ship the chart runtime — bundle size for `/dashboard` came in at 8.62 kB First-Load JS, slightly leaner than `/measurements/history` (6.16 kB) which always loads Recharts.
- `app/page.tsx` — root redirect now points authenticated users at `/dashboard` instead of `/profile` (Phase 7 made `/dashboard` the real landing surface; `/profile` stays reachable from the Settings/profile area).

#### Notes / decisions

- **Pipeline + pure-helper split.** The DELIVERY_PLAN.md Phase 7 task says "use MongoDB aggregation pipelines (group by date) — write these as documented helpers in `lib/agg/`." The trap with that wording is to do *all* the work in Mongo, including the F-DSH-5 ±10% calculation and the days-within-goal count. Instead the pipelines do the heavy "scan and group", and the goal/threshold/average semantics live in pure JS where they're cheap to unit-test without a Mongo. This also means a future tweak to the ±10% rule (say, "±15% on weekends") doesn't require touching any aggregation code.
- **F-DSH-5 toast vs chip.** The chip on the "Remaining" card shows the *resting* state — visible whenever the day's calories are within ±10%. The snackbar is the *transition* notification — shown only when the user crosses from "not met" into "met" while the page is open. This avoids a stale "you reached your goal!" toast popping every time the user reloads a tab six hours later, which is what would happen if it just fired whenever `goalMet === true` on render.
- **Days-within-goal excludes zero-calorie days.** A user who logs no food on a given day shouldn't get credit for "hitting their goal of 0 kcal". The `daysWithinGoal` helper has an explicit `c > 0` guard for this; one of the unit tests pins the behaviour with a `[0, 0, 0]` series and a `2000` goal, asserting `0`. Zero is treated as "no log entered", not as "ate nothing".
- **Today payload uses `Promise.all` for the six reads.** Five aggregations + one `User.findById` for the goal block; in serial these would compound into ~5 round-trips, in parallel they finish in roughly one. NFR-1 (FCP < 2s on 3G, LCP < 2.5s) and the Phase 7 DoD ("Page interactive in <500ms after data loads") are easier to hit when the API itself doesn't sit idle waiting on sequential queries.
- **Recharts only loads on the week/month tabs.** `SummaryCharts` is `dynamic(..., {ssr: false})`-imported by `range-view.tsx`, so opening the dashboard on the Today tab — which is the default and the most common landing — never ships the ~100 kB Recharts runtime. Switching to the Week tab triggers the chunk fetch and a skeleton placeholder fills the chart slot until it lands.
- **Calendar gap-filling lives server-side.** `buildSummaryPayload` walks the window day-by-day and emits a row for every date with zeroes/nulls when the user logged nothing that day. The chart components can then assume an unbroken array — no client-side date arithmetic, no off-by-one risk if the user's TZ disagrees with the server's.
- **`/dashboard` is the new landing.** `app/page.tsx` redirected to `/profile` from Phase 1 because Phase 1 was the only auth'd page that existed. Phase 7's `/dashboard` is the real landing surface now (PRD §8 lists it first), so the redirect is updated. `/profile` remains reachable through navigation; nothing else changes about it.

### Phase 6 — Body Measurements

- `BodyMeasurementEntry` model per PRD §4.7 with the eleven `measurementsCm` sub-fields plus `weightKg`, `bodyFatPercent`, `note`, and the reserved `photos: string[]` (declared but never written in v1 — DELIVERY_PLAN.md Phase 6 says no upload UI). All numeric fields default to `null` so a user can log only what they care about that day (F-BM-1). Compound `(userId, date desc)` index for the cards-page "give me everything sorted newest-first" read pattern; `timestamps: true` so the route can also tiebreak by `createdAt` when two entries land on the same calendar day.
- `lib/models/measurement-enums.ts` — pure constants (`CM_METRICS`, `ALL_METRICS`, `METRIC_LABELS`, `METRIC_UNITS`) so the cards / form / history client components can import them without dragging Mongoose into the bundle (same split pattern as `user-enums`, `food-enums`).
- `lib/measurements/delta.ts` — **the heart of Phase 6's "delta vs previous" semantics**. `summariseMetrics(entries)` returns, per metric, `{latest, previous, delta}` where "previous" is the previous *non-null* value of THE SAME metric — not the second-most-recent entry overall. This is what makes the Phase 6 DoD line "logging only weight one day and only chest the next does not break delta display" actually true: each metric has its own per-metric history, computed independently. `metricSeries(entries, metric)` produces an oldest→newest array suitable for the Recharts `<LineChart>` on the history page. Both helpers are pure (no DB, no React) and accept either ISO-string or `Date` for the `date` field. Deltas are rounded to 3 decimals to avoid IEEE-754 fuzz like `70.2 - 70.1 = 0.10000000000000142`.
- `lib/validation/measurement.ts` — strict Zod for both create and update. Numeric fields are `number().positive().nullable().optional()`; `bodyFatPercent` is `[0, 100]` (clamped per the physical units). Both schemas `.refine()` that **at least one numeric metric must be set** — a measurement entry with just a date and a note is meaningless, so the API 400s rather than persisting an empty doc. Update schema additionally rejects an empty body (`{}` → 400) and is `.strict()` so a sneak-in of `userId` or `photos` (the v1-reserved field) is rejected. Top-level *and* nested `measurementsCm` objects are strict — `{measurementsCm: {ankle: 25}}` fails because "ankle" isn't in PRD §4.7.
- API routes (all `userId`-scoped, error shape `{error, details?}` per NFR-4):
  - `GET /api/measurements?from=YYYY-MM-DD&to=YYYY-MM-DD&limit=N` — sorted `date desc, createdAt desc`. Default cap = 1000 entries, hard cap also 1000 (the cards page needs *all* of a metric's history to compute "previous", but a single user is unlikely to ever exceed this; pagination can come later if it does).
  - `POST /api/measurements` — validates with `MeasurementCreateSchema`, persists `date` as midnight UTC of the user's calendar day (PRD §4.3 / §4.7 convention shared with the other log models). `photos` is forced to `[]` server-side regardless of payload — the v1 reserved field cannot be written.
  - `PATCH /api/measurements/[id]` — cross-user → 403 (project rule #4). Circumferences are **merged**, not replaced, so a PATCH like `{measurementsCm: {chest: 101}}` updates only chest and leaves waist/hips/etc untouched. `markModified("measurementsCm")` is required because Mongoose's nested-doc change detection misses key-by-key writes.
  - `DELETE /api/measurements/[id]` — cross-user → 403.
- Pages (all gated by the existing `withAuth` middleware — `/measurements/:path*` was already in the matcher from Phase 1, so unauth visits redirect to `/login?callbackUrl=…` for free):
  - **`/measurements`** — cards page. `MeasurementsCardsView` calls `summariseMetrics` over the full entry list, renders a `Grid` of one card per metric **that has at least one logged value** (others are hidden so the page stays uncluttered for a new user). Each card shows the latest value, the date it was logged, and a `Chip` with the signed delta (e.g., `−0.8 kg since Apr 15`) or `First entry` when no previous sample exists. The chip uses a `−` minus glyph for negative deltas (`-` would render too thin against `+`); colour is by magnitude (≥0.1 → warning) since up vs down has no universally "good" direction (gaining weight can be a goal). Each card has `data-testid="metric-card-${metric}"` and the chip `data-testid="metric-delta-${metric}"` so the e2e can target them deterministically.
  - **`/measurements/new`** — form page. One Date input plus inputs for `weightKg`, `bodyFatPercent`, all eleven circumferences, and a multiline note. Empty inputs are dropped from the POST body (not sent as `null`), so the user genuinely "logs only what they want today". Client-side guard mirrors the Zod `.refine()`: at least one numeric metric must be set before the request fires. After save, the TanStack Query `["measurements"]` cache is invalidated and the user is redirected back to the cards page so the new value shows up.
  - **`/measurements/history`** — trend chart page. A metric selector (one entry per metric in PRD order) and a 1W / 1M / 3M / 1Y / All toggle group. The selected range becomes a `?from=` query param on `/api/measurements`; "All" omits the param entirely. The chart itself is a Recharts `<LineChart>` rendered through `next/dynamic` with `ssr: false` — Recharts is sizable runtime code that we don't want shipped on the cards page. Empty-state message when no data falls in the selected range. The chart container has `data-testid="history-chart-${metric}"` for the e2e.

#### Notes / decisions

- **"At least one numeric metric" is enforced in Zod, not just in the UI.** F-BM-1 says all numeric fields are optional, but a measurement entry with no numbers (just a date and a maybe-note) is meaningless data. The `.refine()` on `MeasurementCreateSchema` blocks this at the API layer so a bad client (or a future replay endpoint in Phase 8) can't sneak it in. The error message tells the user what they need to provide.
- **Per-metric "previous", not per-entry.** This is the part of the spec that's easy to get subtly wrong. If user logs only weight on Mon and only chest on Tue, then chest's "previous" must be `null` (the user has never logged chest before) — not Monday's weight value. The `summariseMetrics` algorithm walks the entry list in date-desc order and tracks the first two non-null samples *of each metric independently*. Two of the unit tests pin this behaviour explicitly so a future refactor can't regress it.
- **`photos` is reserved but never written.** The schema declares `photos: string[]`, the route forces `photos: []` on POST regardless of payload, and the Update Zod schema is `.strict()` so a PATCH cannot set it either. When the photo upload feature lands (deliberately deferred to the out-of-scope backlog in DELIVERY_PLAN.md), the API surface is already shaped for it.
- **PATCH merges `measurementsCm`, doesn't replace it.** A user updating just chest should not nuke their previously-logged waist value. The route iterates `CM_METRICS` and only touches the keys present in the payload. `markModified("measurementsCm")` is needed because Mongoose's dirty-tracking on nested sub-docs misses single-key writes — without it the save would silently drop the change.
- **Recharts is dynamically imported on the history page only.** It's a non-trivial runtime (≈100 kB minified) that never needs to load on the cards page. `dynamic(..., {ssr: false})` keeps the cards page's bundle lean and matches Phase 0's intent to use Recharts only where charts actually appear.
- **Cards page hides metrics with no data.** A brand-new user logging only weight should see one weight card, not thirteen empty cards with "—" placeholders. When the user logs more metrics over time, more cards appear.
- **No pagination on `GET /api/measurements`.** The cards page genuinely needs the full per-metric history to compute "previous", and a year of daily measurements is ~365 entries — well under the 1000-entry cap. If a power user crosses that, the cap silently limits the read; pagination is a Phase 10 concern (or whenever it actually breaks).

### Phase 5 — Exercise, Water, Sleep Logs

- Three new Mongoose models per PRD §4.4–§4.6, each `userId`-indexed and with a compound `(userId, date)` index for the per-day queries the dashboard (Phase 7) and these pages need:
  - `ExerciseEntry` — required `caloriesBurned` (≥0), nullable `note`, `loggedAt` defaulted to `new Date()`. Multiple per day permitted (F-EX-2 — the dashboard sums them).
  - `WaterLogEntry` — required `amountMl` (≥1), `loggedAt` defaulted to `new Date()`. The PRD field name is preserved verbatim (project rule #3).
  - `SleepEntry` — `bedtime`, `wakeTime`, `durationMinutes` (computed at write time), `quality` ∈ [1,5] integer, nullable `note`. `date` is the wake date (PRD §4.6 explicit choice), midnight UTC, so `?date=YYYY-MM-DD` queries return "the night that ended on this date".
- `lib/log/sleep.ts` — `sleepDurationMinutes(bedtime, wakeTime)` computes duration from two `Date` instants, rounded to the nearest minute. **Cross-midnight just works** because both inputs are full datetimes — no calendar arithmetic, just `(end - start) / 60_000`. Returns `null` when wakeTime ≤ bedtime or either input is invalid; the API turns that into a 400. The route handler computes duration on POST (and recomputes on PATCH if either timestamp changes) so clients can never desync `bedtime`/`wakeTime` from `durationMinutes`.

- API routes (all userId-scoped, error shape `{error, details?}` per NFR-4):

  - `GET /api/logs/sleep?date=YYYY-MM-DD`, `POST /api/logs/sleep`, `PATCH/DELETE /api/logs/sleep/:id`. POST and the timestamp-touching branch of PATCH both run `sleepDurationMinutes` and return 400 if it's null, so the persisted `durationMinutes` is always consistent with its timestamps.

#### Notes / decisions

- **Water has no PATCH endpoint.** PRD §7 lists `DELETE /api/logs/water/:id` only, and the UX matches: a wrong amount is logged again or removed, not edited. This matches the F-WTR-1 quick-add philosophy — entries are atomic mini-events rather than long-form records.
- **Sleep `durationMinutes` is server-computed, not client-supplied.** Even though the form previews it, the API recomputes it from `bedtime` and `wakeTime` on POST and on any timestamp-touching PATCH. - **Sleep `date` = wake date.** PRD §4.6 is explicit; this matches how users think ("how did I sleep last night?" → asks about the date you woke up). It also makes the per-day GET semantics intuitive: `?date=2026-05-03` returns the night that ended this morning.
- **Sleep duration uses real `Date` instants, not local time arithmetic.** That's why crossing midnight Just Works — the difference of two ms-since-epoch is timezone-independent. The unit test for the 23:30 → 07:15 case pins this behaviour so a future regression to "local time of bedtime + delta" would fail loudly.
- **Exercise `note` and sleep `note` Zod allow `null` *and* `undefined`.** Forms send `null` to clear; admin tools that PATCH only one field can omit the key. Both shapes are equivalent server-side (`undefined` means "leave alone", explicit `null` means "clear").

The edit page is a server component that loads the food and 404s for ids that aren't valid ObjectIds *or* belong to another user (UI mirrors the API's 403 by returning 404 to avoid leaking existence).

#### Tests

- Unit (11 new): `__tests__/validation/food.test.ts` covers required fields, serving-unit enum, positive serving size, non-negative calories/macros, nullable fiber/sugar, strict-mode rejection of unknown keys for both create and update schemas.
- E2E:
  - `foods-flow.spec.ts` — full CRUD: create two foods → search "egg" matches one and not the other → favorite toggle reflected on the Favorites tab → edit calorie value → list refreshes with the new value → delete with confirm dialog → row removed.
  - **Second e2e: cross-user 403** — user A creates a food, user B signs in in a fresh context, attempts GET / PATCH / DELETE on A's food id and asserts each returns **403**. Unauth request gets 401. A's food survives intact. This is the first place project rule #4's "tried to read another user's data → 403" test convention is exercised.

#### Migrations / indexes

- New collection: `foods`
  - Compound index `{ userId: 1, name: 1 }` (lookup / search)
  - Compound index `{ userId: 1, lastLoggedAt: -1, timesLogged: -1 }` (frequent-first sort)
  - Compound index `{ userId: 1, isFavorite: 1 }` (favorites filter)
- No data migration; the collection is created lazily on first insert.

#### Notes / decisions

- **Hard delete, no soft delete.** Per F-FOOD-2 + DELIVERY_PLAN.md Phase 3 task list, the snapshot pattern (Phase 4) preserves history on log entries, so cascading the delete to the food doc is correct.
- **Status code for cross-user access: 403, not 404.** Project rule #4 is explicit about 403. This does technically leak existence (404 vs 403 distinguishes "doesn't exist" from "exists but not yours"), but the rule prioritises clarity over enumeration-resistance for this app's threat model.
- **`timesLogged` and `lastLoggedAt` are read-only in this phase.** They're declared on the schema with sensible defaults; Phase 4 will increment them when food logs are created.

### Phase 1 — Auth & User Profile

- `User` Mongoose model implemented per PRD §4.1, with `profile` / `goals` / `reminders` sub-docs and `toJSON` transform that strips `passwordHash` and exposes `id`. Profile fields are optional at signup so users can complete them later on `/profile`; Phase 2 will treat missing values as "needs onboarding" rather than schema errors.
- `PasswordResetToken` collection (separate from `User` to keep PRD §4.1 canonical): stores SHA-256 hash of a 32-byte random token + expiry. TTL index on `expiresAt` lets MongoDB auto-purge expired rows.
- Shared enums extracted into `lib/models/user-enums.ts` so client components can import `ACTIVITY_LEVELS`, `SEX_VALUES`, etc. without pulling Mongoose into the browser bundle.
- Shared Zod schemas (`lib/validation/auth.ts`, `lib/validation/profile.ts`) used on both client (RHF resolver) and server (route validation) per project rule #6.
- API error responses use the `{ error: string, details?: object }` shape (NFR-4) via `lib/api/errors.ts`.
- NextAuth configured with the Credentials provider, JWT session strategy, and a typed `session.user.id` / `session.user.role` (augmentations in `types/next-auth.d.ts`). bcrypt cost factor 12 (F-AUTH-1).
- Email sending via `nodemailer`: `jsonTransport` in dev (logs payload to the server console); SMTP env vars (`SMTP_HOST`, `SMTP_PORT`, `SMTP_SECURE`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM`) take over when set.
- API routes:
  - `POST /api/auth/signup` — creates user with bcrypt-hashed password.
  - `POST /api/auth/forgot-password` — generates a 24-hour reset token, persists hash, sends email; always returns `{ ok: true }` to avoid leaking which emails exist.
  - `POST /api/auth/reset-password` — verifies token hash + expiry, updates `passwordHash`, marks token used.
  - `GET / PATCH /api/profile` — session-scoped read/update; never accepts a userId from the client.
- Role-based middleware (`middleware.ts`) using `next-auth/middleware`'s `withAuth` to gate `/profile`, `/dashboard`, `/log/*`, `/foods/*`, `/measurements/*`, `/settings/*`. F-AUTH-5 ✓.
- Auth UI: `/signup`, `/login`, `/forgot-password`, `/reset-password` (RHF + Zod, MUI). After signup, the user is auto-signed-in and lands on `/profile`.
- Profile page (server-side session check + DB read; client form for editing) including units sub-form and a sign-out button (F-AUTH-4).
- `/` redirects to `/profile` if authenticated, else `/login` (PRD §8 IA).

#### Tests

- Unit:
  - `__tests__/auth/password.test.ts` — bcrypt cost 12, hash/verify, salt randomness.
  - `__tests__/auth/tokens.test.ts` — token entropy, deterministic SHA-256 hash, **24h expiry verified with mocked clock** (`vi.useFakeTimers()` + `vi.setSystemTime`).
  - `__tests__/validation/auth.test.ts` — Signup/Login/Forgot/Reset Zod schemas, edge cases.
- E2E:
  - `e2e/smoke.spec.ts` — `/` redirects to `/login` for unauth visitors.
  - `e2e/auth-flow.spec.ts` — signup → profile edit → reload (session persists) → sign out → sign back in. Plus a middleware-redirect test.

#### Migrations / indexes

- New collection: `users` — unique index on `email`; secondary index on `role`.
- New collection: `passwordresettokens` — index on `userId`, index on `tokenHash`, **TTL index on `expiresAt`** (auto-purge).
- `lib/db.ts` now connects to the `caloriespro` database explicitly (overridable via `MONGODB_DB`); Atlas URIs without a path no longer fall through to `test`.

#### Notes

- The "tried to read another user's data → 403" test required by project rule #4 is N/A for Phase 1: `GET/PATCH /api/profile` derive the user ID from the session and never accept a path/query userId, so there is no cross-user attack surface here. Cross-user 403 tests start in Phase 3 (`/api/foods/:id`) and onwards.
